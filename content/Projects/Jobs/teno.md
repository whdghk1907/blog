
**OpenDRIVE API Module 개발 개요**

OpenDRIVE API 모듈은 향후 Web Editor 개발 완료 시 사용될 예정이며, 현재 프로토타입이 구축된 상태입니다. 이 모듈은 기존의 Opendrive 변환 작업에 사용되던 Python 코드를 FastAPI 서버에 올려 재사용성을 높이고 작업 효율을 극대화하려는 목표를 가지고 있습니다.

### 기존 문제와 개선 사항

- **변환 중 비효율성**: 기존 mgeo_editor에서는 Opendrive 변환이 진행되는 동안 다른 작업을 수행할 수 없어 비효율적인 시간 소모가 발생했습니다. 이를 서버 기반으로 전환하여 맵 작업의 효율성을 개선했습니다.
- **코드 관리**: mgeo_editor와 Opendrive 변환 코드를 분리하여 독립적으로 관리함으로써 코드 품질을 높이고 유지 비용을 절감했습니다.

### 주요 작업 내용

1. **pylib_opendrive 레포지토리 생성 및 submodule 분리**: 독립적 코드 관리 환경을 위한 레포지토리를 생성하고 submodule로 분리했습니다.
2. **opendrive convert 작업 분산 워크플로우 작성**: opendrive 변환 과정은 3~10분 이상의 비용을 소모하는 큰 규모의 요청으로,이를 효과적이면서 병렬로 처리하기 위해 프레임워크 celery, redis, k8s등을 테스트하고 도입하여 새로운 워크플로우를 도입했습니다.

### 향후 계획

현재 단독 사용자용 프로토타입은 완성되었으며, 동시 10~20명 사용자를 지원할 수 있는 수준으로 확장 작업 중입니다. GUI와 API 서버를 통합하는 아키텍처는 QGIS나 ArcGIS와 같은 시장 내 주요 소프트웨어에서 활용하는 구조로, 경쟁력을 강화하고자 이를 도입했습니다.

특히, Morai 내에서는 GIS 로직을 서버에서 구동한 경험이 없어, 이를 처음으로 시도하며 GUI 통합의 장점과 서버의 적정 규모 및 기술 스택에 대해 검토하고 있습니다.

**OpenDRIVE API Module 배포 업무**

OpenDRIVE API 서버 배포와 CI/CD 작업을 위해 Docker, Jenkins, GitLab Runner, K8s 등의 기술을 도입하여 아키텍처를 구성했습니다. 기존에는 PC가 재부팅되거나 프로그램이 종료될 때마다 개발자가 수동으로 재시작하고 배포해야 했으나, 이를 개선했습니다.

### 주요 개선 사항

- **자동화된 배포 환경**: Jenkins와 GitLab Runner를 Docker에 배포하고, 필요한 설정을 Dockerfile, Shell Script, .yaml 파일에 정리하여 수작업 비용을 최소화했습니다.
- **효율적 요청 처리**: OpenDRIVE 변환 작업은 3~10분의 시간이 소요되므로, CPU 사용량에 따라 복사본을 자동 배포하는 Kubernetes HPA 기능을 적용하고, Celery Queue 프레임워크로 요청을 순차적으로 분배하는 구조를 테스트 중입니다.

### 보안 및 최적화

- **코드 암호화**: AWS와 같은 클라우드 환경에 배포 시 소스 코드 보호를 위해 Cython을 활용한 프로젝트 단위 암호화 스크립트를 작성하고, 이를 CI/CD 과정에 포함했습니다.
- **코드 품질 검사**: 향후 PEP8, Black 등의 도구를 활용하여 코드 문법 검사 자동화를 도입할 예정입니다.

**Map Editor 기능 개발 업무**

팀 내부 및 외부의 요청, 또는 필요에 의해 Map Editor의 기능을 개발하고 배포했습니다.

### 주요 기능 개발 내용

1. **교차로 내 충돌 가능한 링크 목록화**
   - Agent 팀의 요청으로, Road 간의 충돌 정보를 속성에 제공하는 기능을 개발했습니다.

2. **set L/R Lane Mark 기능 고도화**
   - 기존 로직이 부정확하여, 좌/우 Lane의 방향성과 위치를 기준으로 자신의 Lane을 구분하고 저장할 수 있는 정확한 로직을 새로 개발했습니다.

3. **곡선 계수 정보 저장**
   - 자율주행용 MGeo 데이터를 곡선화할 수 있도록, OpenDRIVE geometry 시스템을 활용하여 spline 곡선화 계수를 line에 저장하는 기능을 구현했습니다.

4. **속성 명칭 공통화 작업**
   - 클라이언트마다 속성 명칭이 달라 정보 파악이 어려운 문제를 해결하기 위해, MgeoPropertyConverter 모듈을 개발하여 속성을 편집할 수 있는 widget을 제공했습니다.

### 기타 기능 개선

- 도로 안내 표지판 데이터 업데이트 코드 정리 및 기능 배포.
- 글로벌 좌표와 local 좌표 선택 표출 기능 개발.

### 향후 계획

VWorld API를 사용하여 배경지도 생성하기, Slice Road 기능을 고도화하기 등
Map Editor 기능 향상을 위한 업무들이 남아있습니다.

**Map Editor 리팩토링 및 오류 수정 업무**

기존 기능에서 발생하는 오류 수정과 MGeo Editor의 구조적 보완 및 리팩토링 작업을 수행했습니다.

### 주요 오류 수정 및 리팩토링 내용

1. **idx 중복 방지**
   - 기존 MGeo Editor에서는 idx 중복 생성을 허용했으나, 중복 객체를 방지하도록 자료 객체 구조를 리팩토링했습니다. 중복된 객체가 존재할 경우 사용자에게 위치를 알리고 Load를 실패하게 하였습니다.

2. **SliceRoad 기능 오류 수정**
   - 동일한 road에 대해 2번 SliceRoad 기능을 수행할 때 idx 중복 오류가 발생하던 문제를 수정했습니다.

3. **프로젝트 전반의 문법 오류 수정**
   - import 없이 선언된 모듈, init 파일 없이 생성된 Python 패키지, 선언되지 않은 입력값을 사용하는 함수 등 프로젝트 내에 남아 있던 문법적 오류들을 수정했습니다.

### 향후 계획

특정한 경우에 Link등 객체들이 display 오류를 발생시키는 오류를 수정할 예정입니다.
수집한 데이터의 z값과 사용자가 직접 생성한 z값이 상이하는 문제가 발생하여 opendrive 파일에서 오류가 발생하는 현상을 수정할 예상입니다.

1. opendrive 로직 최적화
- 현재 opendrive 로직은 로직 난이도에 비하여 소모되는 비용이 크다고 생각합니다. 최적화가 진행될 여지를 조사하고 실행시간을 단축해보는 시간을 가져보고 싶습니다.

2. vworld api 활용
- vworld api에서 제공하는 배경지도 타일링 서비스를 활용하여, 실제 환경과 mgeo 객체를 비교하며 맵을 편집하는 기능을 개발해보는 시간을 가져보고 싶습니다.

3.opendrive api 모듈 고도화
- opendrive api 모듈은 단순히 converting 로직을 수행하는 것 이상으로, 데이터 수집과 관리측면에서 이점을 가져올 수 있습니다. 기회가 된다면 회의를 통해 고도화 계획을 가져보고 싶습니다.
